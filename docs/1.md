3.1 左侧工作台：AI 决策状态显化面板 (AI Reasoning Manifestation)
功能描述： 当右侧手机预览端渲染完成时，左侧工作台下方动态亮起一组彩色胶囊标签。这些标签不允许用户手动点击勾选，它们是 AI 决策结果的“逆向具象化展示”。

标签矩阵库（按大模型实际触发状态高亮）：

版式控制类： [🎯 大转盘视觉 C 位]、[🚗 车身退为科技背景]、[📖 触发横滑杂志画册流]

风格调性类： [💎 注入新豪华绿金调]、[🔥 开启高频裂变色系]、[☕ 匹配接地气 Fellow 话术]

用户价值： Fellow 看着亮起的标签，瞬间能看懂 AI 的排版逻辑：“哦！原来大模型因为我说要‘吸引人’，所以自动帮我把转盘放大了。”

3.2 左侧工作台：多轮语义追问框 (Multi-turn Prompt Workspace)
功能描述： 首版 H5 生成后，下方解锁一个“追问/微调输入框”。

交互逻辑： Fellow 无需调整代码，通过对话向 AI 下达修正指令（例如：“GX 是商务品鉴，别搞得像大卖场抽奖，把车放回顶部放大，转盘放到底部收敛一点。”）。

4. 后端 API 与上下文状态契约 (Context-Aware API)
为了支持多轮追问和微调，/api/generate 接口需由“无状态”升级为“上下文感知状态”。

4.1 请求体 (Request Body) 升级
当前端进行第二轮追问时，POST 请求必须同时携带上一版的完整 JSON 结构与用户的新追问文本：

JSON
{
  "isMultiTurn": true,
  "previousJson": {
    "theme": "pop_active",
    "layoutMatrix": { "componentOrder": ["lucky_wheel", "car_stage"] }
    // ... 第一版的其余完整数据
  },
  "currentInput": "GX是商务品鉴，别像大卖场，把车放大放回顶部，转盘放低收敛。"
}
4.2 后端 System Prompt 强控制逻辑
在后端构造函数中，注入多轮更新的防御性指令：

Plaintext
【重要：多轮增量修改规范】
当接收到 isMultiTurn 为 true 的请求时，你扮演一个“UI重构架构师”。
1. 不要完全推翻上一版数据（previousJson），保持销售已满意的核心营销文案。
2. 深度解析用户的 currentInput（新追问）。如果用户嫌转盘太抢眼，你必须重组 layoutMatrix.componentOrder 序列，将 "lucky_wheel" 移向尾部，并将其组件权重设为收敛。
3. 如果语气/车型需要调整，同步修正 theme 皮肤标签。
4. 严格且仅返回修改后的纯 JSON 矩阵对象。
5. 右侧渲染层：Framer Motion 编排的“液态重组” (Fluid UI)
所有原子组件必须包裹在 <motion.div layout /> 容器中。

动画表现： 当用户发送追问指令，AI 返回调整后的新矩阵后，右侧手机端组件绝不能生硬闪烁切换。大转盘在物理引擎的插值计算下，会顺畅、丝滑地从顶部‘滑落’到屏幕最下方并平滑缩小；与此同时，顶部车身舞台优雅地扩张并放大放光芒。

这种组件在屏幕上动态换位、缩放的“液态流动感”，是证明 AI 真正作为“总设计师”介入 UI 排版的最高视觉证据。
